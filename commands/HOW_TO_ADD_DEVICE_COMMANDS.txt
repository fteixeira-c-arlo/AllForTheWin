How to add device commands (current architecture)
=================================================

Device catalogs are JSON files under commands/. Shared “abstract” workflows
(version, factory reset, firmware update steps, etc.) are defined once in
abstract_command_definitions.json; each device maps its own shell/CLI steps
to those names. No Python changes are required to add a new device line—only
JSON plus a camera model entry (see below).


1. Create commands/<device>_commands.json
-----------------------------------------

Use the same overall shape as e3_wired_commands.json: a top-level object with
a required "commands" array. Optional top-level keys (source, url, models,
etc.) are fine for documentation; only "commands" is required for loading.

Each command object should include:

  name         — Unique id for this step (used when resolving abstract sequences)
  shell        — Full line to run on the device (may be "" for app-handled cmds)
  syntax       — What the user sees in help / UI
  description  — Short help text
  category     — Used to group entries in the Advanced section (e.g. arlocmd, mfg)

For commands that participate in a shared abstract workflow, add:

  abstract     — String matching the "name" field of an entry in
                 abstract_command_definitions.json (e.g. "version", "factory reset",
                 "update url"). Spelling and spacing must match that file.

Entries without an "abstract" field are not tied to the abstract layer; they
appear only under Advanced (help / GUI), while abstract-linked
entries are still in the full catalog for execution but are omitted from the
Advanced list (they run as part of the abstract command’s sequence).


2. Register the profile in commands/command_profiles.json
---------------------------------------------------------

Add a key for your profile id (referenced from camera_models):

  "your_profile_id": {
    "label": "Short label",
    "commands_file": "your_commands.json"
  }

The built-in profile "none" uses "commands_file": null — no device catalog;
only shared system commands apply.

Optional "notes" on an entry is fine (see existing entries).


3. Point models at the profile in commands/camera_models.py
---------------------------------------------------------

On each relevant CAMERA_MODEL_GROUPS entry, set:

  "command_profile": "your_profile_id"

Use the same keys you use in command_profiles.json. Also set
supported_connections, fw_search_models, default_settings, etc., as for
existing groups so the device picker and firmware search behave correctly.


4. Adding a new abstract command (all devices that should support it)
---------------------------------------------------------------------

Abstract commands are global definitions in abstract_command_definitions.json.
Each entry includes at least:

  name                   — User-facing name (what users type / see)
  description            — Help text
  args                   — Argument names for validation
  sequence               — List of device command "name" values to run in order
  transport_restriction  — null or a restriction for connection type

Steps in "sequence" must match the "name" field of entries in each device’s
JSON (the loader resolves abstract runs by those names).

To support the new abstract on a device line:

  • Add the abstract definition to abstract_command_definitions.json.
  • In that device’s commands JSON, ensure every step name in "sequence" exists
    with the correct "shell" (and other fields).
  • On each step that should be hidden from Advanced and driven only via the
    abstract workflow, set "abstract" to that abstract’s "name".

No Python is required for the abstract definition itself; execution is handled
by the existing abstract dispatcher.
